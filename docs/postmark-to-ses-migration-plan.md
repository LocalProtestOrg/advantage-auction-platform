# Postmark → Amazon SES Migration Plan (SMTP mode)

*Planning only — no code. Goal: replace Postmark (account rejected) with Amazon SES via **SMTP mode**, preserving all email functionality: account verification, password reset, outbid, winning-bidder, seller agreement emails, operational close emails, final-report PDF attachments, and future governance notifications. Sending domain: **advantage.bid** (DNS host: `directorysecure.com`).*

## 1. Architecture overview
Move all outbound email onto **Amazon SES using the SMTP interface** (`email-smtp.<region>.amazonaws.com:587`, STARTTLS). Rationale (from the migration assessment):
- The Postmark-specific code is **one file** (`src/services/emailService.js`, currently a raw HTTPS call to `api.postmarkapp.com` with `X-Postmark-Server-Token`). Rewriting its internals to **nodemailer-over-SES-SMTP** keeps the public `sendEmail({to,subject,html,text})` signature unchanged — **all 4 HTTP callers are untouched**.
- The two other senders (`operationalCloseEmailService.js`, `pdfGenerationService.js`) are **already nodemailer/SMTP** and provider-agnostic — they migrate by **env config only** (point `SMTP_*` at SES), no code change.
- **One credential set (SES SMTP) unifies all three senders.** No template changes (all bodies are app-side strings; Postmark Templates API is not used).
- `notificationService._sendEmail` is a **mock** (console.log) — not a real sender; out of scope.

## 2. Current email topology
| Component | Mechanism today | After migration |
|---|---|---|
| `src/services/emailService.js` | Postmark HTTP API (`X-Postmark-Server-Token`, `MessageStream`) | **Rewrite internals → nodemailer over SES SMTP** (same `sendEmail` signature) |
| `notificationWorker.js` → `sendEmail` | account verification, password reset, outbid, winning, reminders, governance notifications | unchanged (calls `sendEmail`) |
| `agreementService.js` → `sendEmail` | agreement signing-link emails | unchanged |

| `operationalCloseEmailService.js` | nodemailer SMTP (`SMTP_HOST/PORT/USER/PASS`) | **env only** → SES SMTP |
| `pdfGenerationService.js` | nodemailer SMTP **+ PDF attachment** | **env only** → SES SMTP (nodemailer handles attachments natively) |
| `notificationService._sendEmail` | mock (console.log) | unchanged (not a real sender) |
| `notifications_queue` + worker retry | DB-queued; failed sends retried | unchanged — **resilience/rollback safety net** |

**Email env vars in use:** `SMTP_PASS` (today = Postmark token *and* SMTP password — dual-purposed), `SMTP_USER`, `SMTP_HOST`, `SMTP_PORT`, `SMTP_SECURE`, `SMTP_FROM`, `EMAIL_FROM`, `EMAIL_REPLY_TO`. (Adjacent, not transport: `ADMIN_EMAIL`, `SITE_URL`/`FRONTEND_URL`/`PUBLIC_BASE_URL`.)
**After migration (SES SMTP):** same names — `SMTP_HOST=email-smtp.<region>.amazonaws.com`, `SMTP_PORT=587`, `SMTP_SECURE=false` (STARTTLS), `SMTP_USER`/`SMTP_PASS` = SES SMTP credentials; `EMAIL_FROM`/`EMAIL_REPLY_TO` unchanged.

## 3. SES setup checklist (summary — full version in §9)
1. AWS account + SES in the chosen region (**us-east-1** recommended).
2. Verify the **advantage.bid** domain identity.
3. Enable **Easy DKIM** → add **3 CNAME** records to `directorysecure.com`.
4. Configure a **custom MAIL FROM** subdomain (e.g. `bounce.advantage.bid`) for DMARC alignment (MX + SPF).
5. Add/adjust **SPF** and **DMARC** on `advantage.bid`.
6. Create **SES SMTP credentials**.
7. **Request production access** (exit the SES sandbox).
8. Set sending quota expectations + bounce/complaint handling (SNS optional).

## 4. DNS requirements (zone: advantage.bid @ directorysecure.com)
> All records are added at the **same DNS host already confirmed editable** (`ns1/ns2.directorysecure.com`).

**Add for SES:**
- **Easy DKIM (3 records):** CNAME `<token1>._domainkey.advantage.bid` → `<token1>.dkim.amazonses.com` (and token2, token3). *(Values generated by SES — copy exactly. These also serve as domain verification in SESv2.)*
- **Custom MAIL FROM** (recommended; subdomain e.g. `bounce.advantage.bid`):
  - MX: `bounce.advantage.bid` → `feedback-smtp.<region>.amazonses.com` (priority **10**)
  - SPF TXT: `bounce.advantage.bid` → `v=spf1 include:amazonses.com ~all`
- **SPF on the From domain** — current record is `v=spf1 +a +mx +ip4:66.147.230.95 include:spf.mtasv.net ~all`. Plan: **add `include:amazonses.com`** and (after cutover) **remove `include:spf.mtasv.net`** (Postmark). Keep within the 10-DNS-lookup SPF limit.
- **DMARC** (recommended): TXT `_dmarc.advantage.bid` → `v=DMARC1; p=none; rua=mailto:<ops-inbox>; fo=1` (start at `p=none` for monitoring; tighten later).

**Remove after cutover is stable (Postmark leftovers):**
- DKIM TXT `20260513133639pm._domainkey.advantage.bid`
- Return-Path CNAME `pm-bounces.advantage.bid` → `pm.mtasv.net`
- `include:spf.mtasv.net` from the SPF record

> ⚠️ Do **not** remove Postmark records until SES is fully verified + cutover complete (avoid a no-email gap).

## 5. Production-access request guidance
SES starts in **sandbox** (send only to verified identities; ~200 emails/day; 1 msg/sec). To send to real users you must **request production access** (AWS Account dashboard → "Request production access"). Approval is usually <24h. Provide: mail type (**Transactional**), website/use case, recipient opt-in description, and your **bounce/complaint handling** process. Suggested wording in §9.7. **Do not announce/enable seller-facing email until production access is granted** and DKIM shows **verified**.

## 6. Migration phases (staging-first, env-gated)
- **Phase 0 — AWS/SES setup (gating):** complete §9 (verify domain, Easy DKIM, MAIL FROM, SMTP creds, production access).
- **Phase 1 — Code (one file):** rewrite `emailService.sendEmail` to send via **nodemailer over SES SMTP** — same `{to,subject,html,text}` inputs, same `EMAIL_FROM`/`EMAIL_REPLY_TO` behavior, keep the "skip when unconfigured" guard, drop the Postmark header/`MessageStream`. *(Optional later: add attachment support to `emailService` and route the two nodemailer services through it — not required now.)*
- **Phase 2 — SMTP services:** point `SMTP_HOST/PORT/USER/PASS/SECURE` at SES on the service env; the two nodemailer senders work unchanged.
- **Phase 3 — Staging validation:** set SES SMTP env on the **staging** service; run §7. Confirm SPF/DKIM/DMARC **pass** in received headers and SES shows the domain **verified + DKIM active**.
- **Phase 4 — Production cutover:** add SES env to the **prod** service; deploy the one-file change; smoke-test; monitor SES console (sends/bounces/complaints) + Railway logs; after stable, remove Postmark records (§4) and decommission the Postmark token.

## 7. Validation plan
Run on staging first, then a smoke subset on prod (seeded/internal recipients only; clean up):
1. `POST /api/admin/email/test` → delivered; inspect headers for **SPF=pass, DKIM=pass, DMARC=pass**.
2. Exercise each functional type: **account verification, password reset, outbid, winning-bidder, agreement signing-link, governance notifications (return-to-draft/rejected), operational close email, final-report PDF attachment**.
3. Verify the **PDF attachment** arrives intact (the nodemailer/SES path).
4. Confirm **`notifications_queue`** drains (worker marks sent) and that a forced failure **retries** (no loss).
5. Check SES console: send count, **bounce rate <5%, complaint rate <0.1%** (AWS thresholds).
6. Confirm **agreement signing links** resolve (`PUBLIC_BASE_URL`/`FRONTEND_URL` correct) — ties to the prod env blocker.

## 8. Rollback plan
- **Code:** the change is isolated to **`emailService.js`** (+ env). Revert that commit + redeploy to restore prior behavior.
- ⚠️ **Postmark is NOT a rollback target** (account rejected). Rollback options:
  1. **Fail-safe by design:** notifications persist in **`notifications_queue`** and the worker **retries** — a transient SES issue **queues, doesn't lose** mail; `emailService` returns `{skipped:true}` when unconfigured instead of crashing.
  2. **Backup sender (recommended):** provision a **second SES region** or a fallback SMTP relay behind the same `SMTP_*` env so rollback = repoint env, not redeploy.
- **Data:** none affected — email is stateless transport; queued notifications survive an outage.

## 9. AWS SES onboarding checklist — advantage.bid

### 9.1 AWS account requirements
- An AWS account with billing enabled and **root MFA**; create a dedicated **IAM user/role** for SES (least privilege: `ses:SendRawEmail`, `ses:SendEmail`) — do **not** use root for sending.
- Decide where SES SMTP credentials/secrets live: **Railway service env vars** (`SMTP_USER`/`SMTP_PASS`), never committed (per the project security rules).
- Budget/alarms: a CloudWatch billing alarm; SES is pay-per-email (very low cost).

### 9.2 SES region recommendation
- **Recommended: `us-east-1` (N. Virginia)** — broadest SES feature support, common default, and aligned with the platform's us-east infra. The region fixes the **SMTP endpoint** (`email-smtp.us-east-1.amazonaws.com`) and the **DKIM CNAME targets** (`...dkim.amazonses.com`) and **MAIL FROM MX** (`feedback-smtp.us-east-1.amazonaws.com`). Pick one region and keep all identities/credentials there. *(A second region later = the backup-sender rollback option.)*

### 9.3 Identity verification (domain)
- SES → **Verified identities → Create identity → Domain → `advantage.bid`**.
- Choose **Easy DKIM** (below) — in SESv2 the DKIM CNAMEs double as domain verification (no separate `_amazonses` TXT needed for the modern flow; if the console shows a verification TXT, add it too).
- **From/Reply-To (decided):** `EMAIL_FROM = notifications@advantage.bid` (primary sender, aligns DKIM/DMARC); `EMAIL_REPLY_TO = info@advantage.bid` (or `support@advantage.bid`) — replaces the legacy gmail reply-to. The verified domain identity covers any `@advantage.bid` address.

### 9.4 Easy DKIM setup
- In the identity, enable **Easy DKIM** (RSA 2048).
- SES generates **3 CNAME** records: `<token>._domainkey.advantage.bid → <token>.dkim.amazonses.com` (×3).
- Add all 3 **CNAME** records to the `directorysecure.com` zone **exactly** (no trailing-dot/double-domain mistakes — the same registrar-formatting class of error that can break DKIM).
- Wait for SES to show **DKIM: Verified / Successful** (propagation minutes–hours). DMARC alignment requires DKIM pass on the **From** domain — Easy DKIM provides this.

### 9.5 Custom MAIL FROM (recommended)
- Set **MAIL FROM domain** = `bounce.advantage.bid` (a subdomain).
- Add: **MX** `bounce.advantage.bid → feedback-smtp.us-east-1.amazonaws.com` (priority 10) and **SPF TXT** `bounce.advantage.bid → v=spf1 include:amazonses.com ~all`.
- Benefit: SPF aligns on the MAIL FROM subdomain → stronger DMARC. (Without it, SES uses `amazonses.com` as MAIL FROM and SPF won't align to `advantage.bid`, but DKIM alignment via Easy DKIM still satisfies DMARC.)

### 9.6 SMTP credential creation
- SES → **SMTP settings → Create SMTP credentials** (creates an IAM user; SES converts its secret access key into an **SMTP password** — shown once, copy immediately).
- Record: **SMTP endpoint** `email-smtp.us-east-1.amazonaws.com`, **port 587** (STARTTLS), **username** + **password**.
- Set Railway env: `SMTP_HOST=email-smtp.us-east-1.amazonaws.com`, `SMTP_PORT=587`, `SMTP_SECURE=false`, `SMTP_USER=<smtp-username>`, `SMTP_PASS=<smtp-password>`, `EMAIL_FROM=notifications@advantage.bid`, `EMAIL_REPLY_TO=info@advantage.bid` (or `support@advantage.bid`). (Never commit these.)

### 9.7 Production-access request wording (template)
> **Mail type:** Transactional.
> **Website URL:** https://advantage.bid (Advantage Auction marketplace).
> **Use case:** Transactional emails to registered users and sellers of our online auction platform — account verification, password reset, outbid and winning-bidder notifications, seller agreement (e-signature) links, and auction operational notices. All recipients are authenticated users who created an account or were explicitly onboarded; we do not send marketing/bulk mail from this identity.
> **List/opt-in:** Recipients are our own registered users; addresses are collected at signup/onboarding. No purchased lists.
> **Bounce & complaint handling:** We monitor SES bounce/complaint metrics (and SNS notifications); hard-bounced and complained addresses are suppressed from further sends; our notification worker retries transient failures and stops on permanent failures. We honor unsubscribe/contact-support requests.
> **Expected volume:** low transactional volume (tens–hundreds/day initially), well within standard limits.

Submit via **SES → Account dashboard → Request production access**; expect a decision typically within 24 hours.

### 9.8 Post-setup verification
- SES identity: **Verified**, DKIM **active**, MAIL FROM **verified**.
- Account: **out of sandbox** (production access granted), sending quota adequate.
- Send a test from staging → headers show **SPF/DKIM/DMARC = pass**.
- Only then enable seller-facing email and proceed to the production cutover (Phase 4).
